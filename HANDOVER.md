# 项目交接指南 (Handover Document) - Bid 项目

你好，Antigravity IDE 的 AI 助手。我是上一任负责此项目的 Gemini CLI 助手。为了让你能快速接手并继续推进“智能标书生成助手 (Bid)”的开发，我为你梳理了目前的项目状态、核心资产以及待执行的路线图。

## 1. 项目核心愿景
本项目旨在通过 **Gemini Pro (主脑/Commander)** 与 **Dify Workflow (执行者/Executor)** 的双层架构，实现标书从“解析大纲”到“并发生成内容”再到“还原 Word 格式”的自动化闭环。

## 2. 核心架构资产 (核心共识)

### 2.1 双层模型协作模式
- **Commander (FastAPI 端)**：负责全局逻辑。它解析 Word 目录，将其转化为任务列表，并按优先级并发调度 Dify。
- **Executor (Dify 端)**：负责章节内容填充。它输出严格遵循“多模态内容协议”的 JSON，包含文本、Markdown 表格和 Mermaid 流程图。

### 2.2 多模态内容协议 (Critical)
Dify 返回的必须是如下格式，这是后端解析器 (`python-docx`) 能还原 Word 格式的前提：
```json
{
  "section_id": "章节号",
  "segments": [
    { "type": "text", "value": "..." },
    { "type": "chart", "format": "mermaid", "value": "..." },
    { "type": "table", "format": "markdown", "value": "..." }
  ]
}
```

## 3. 当前开发进度与现状

### 3.1 已完成 (Success)
- [x] **后端解析引擎**：基于 `python-docx` 的高精度大纲提取算法，可识别 `outlineLevel` 和特定正则表达式，完美支持复杂银行标书。并已修复提取规则，支持长文本过滤和中文标题样式。
- [x] **前端 UI 雏形**：使用 Next.js + Tailwind 构建，具备目录树渲染、上传状态展示和 AI 对话侧边栏。
- [x] **工程标准化**：项目已实现 Docker 化，`docker-compose.yml` 稳定配置了前后端环境（已修复本地和容器内的端口冲突问题）。
- [x] **并发调度逻辑**：后端 `main.py` 已完成向 Dify 发起并发请求的具体逻辑（基于 `asyncio.Semaphore` 和 `httpx.AsyncClient`）。
- [x] **Dify 集成与预处理**：`/generate` 接口已连通 Dify Workflow。并在调用前引入了 `Flash 模型预处理机制`，能够提前将简单的上下文重写为详尽的指令以指导较为机械的执行模型；同时后端完美拦截并正则剔除了大模型可能产生的 `<think>` 标签输出。
- [x] **多模型智能调度**：在 Settings 中引入了动态模型下发。大纲理解采用高智力 `Pro` 模型，高频短任务采用高性价比 `Flash` 模型。支持实时拉取 Gemini 官方模型列表供前端选择。
- [x] **数据持久化底座**：引入 SQLite 和 SQLAlchemy，实现前端页面刷新后状态自动恢复。
- [x] **并发调度与状态交互联动**：前端支持“点击父节点自动提取并队列化所有叶子节点”，并实装了右下角的抽屉（Drawer）悬浮面板，可视化监控并发进度。
- [x] **多模态还原引擎**：后端使用混合解析器（支持严格 JSON 或自动降级为纯 Markdown 解析）。成功将 Markdown 文本、原生表格以及经由 `kroki.io` 渲染的 Mermaid 图表汇编为 `.docx` 文件；前端工作区接入 `react-markdown` 实现了富文本沉浸式预览。

## 4. 你的下一步任务 (Next Actions - 版本：批量完成)

目前项目已达成“批量完成”里程碑，并发调度和状态恢复的基础设施已非常稳固。但在实际业务中，**Dify 引擎的集成目前是最大的瓶颈**，存在较大程度的不可用。接下来的工作必须重点围绕 Dify 展开：

1. **Dify 输入端可用性重构 (Input Usability)**：
   - 优化发往 Dify 的 Prompt 结构，确保 Dify 能够准确理解章节所处的上下文层级。
   - 考虑在前端引入更多结构化的配置项（如预置不同行业的系统提示词），甚至是对 Dify 请求 payload 的灵活定制。
2. **Dify 输出端与 Word 还原引擎的深度对齐 (Output Conversion)**：
   - 彻底解决 Dify 输出 JSON 与 `python-docx` 还原逻辑之间的兼容性问题（目前存在较大程度的格式丢失或解析失败）。
   - 建立更强大的“兜底机制”和数据清洗层，确保即使模型输出残缺的 JSON 或复杂的 Markdown 表格/图表，也能 100% 安全且完美地插入到生成的 `.docx` 文件中。
3. **多级目录拖拽排序与知识库 (RAG) 融合**：
   - 在前端补充目录拖拽排序。
   - 探索 Dify 知识库 (RAG) 接口的传参规范，让生成的每一章节内容都能引用真实的业务文档库。
