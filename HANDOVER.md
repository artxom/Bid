# 项目交接指南 (Handover Document) - Bid 项目

你好，Antigravity IDE 的 AI 助手。我是上一任负责此项目的 Gemini CLI 助手。为了让你能快速接手并继续推进“智能标书生成助手 (Bid)”的开发，我为你梳理了目前的项目状态、核心资产以及待执行的路线图。

## 1. 项目核心愿景
本项目旨在通过 **Gemini Pro (主脑/Commander)** 与 **Dify Workflow (执行者/Executor)** 的双层架构，实现标书从“解析大纲”到“并发生成内容”再到“还原 Word 格式”的自动化闭环。

## 2. 核心架构资产 (核心共识)

### 2.1 双层模型协作模式
- **Commander (FastAPI 端)**：负责全局逻辑。它解析 Word 目录，将其转化为任务列表，并按优先级并发调度 Dify。
- **Executor (Dify 端)**：负责章节内容填充。它输出严格遵循“多模态内容协议”的 JSON，包含文本、Markdown 表格和 Mermaid 流程图。

### 2.2 多模态内容协议 (Critical)
Dify 返回的必须是如下格式，这是后端解析器 (`python-docx`) 能还原 Word 格式的前提：
```json
{
  "section_id": "章节号",
  "segments": [
    { "type": "text", "value": "..." },
    { "type": "chart", "format": "mermaid", "value": "..." },
    { "type": "table", "format": "markdown", "value": "..." }
  ]
}
```

## 4. 你的下一步任务 (Next Actions - 生产力聚焦)

根据当前项目的里程碑，下一阶段（阶段三）的核心目标是**提示词质量与生成结果的规整优化**。
当你再次启动时，如果用户说出类似“开始执行下一阶段的提示词优化”或“立刻开始工作”，请直接从以下三点着手进行代码和系统配置修改：

1. **禁用内部自编号与大纲层级展开**：我们规划给大模型的内容节点都已经是最低级别的叶子节点了。必须在提示词中严格禁止大模型在正文中自行带有“第一节”、“第二节”或“1.3”、“1.4”这样的章节标号。
2. **强化 Flash 预处理模型的“审慎”原则**：Flash 这一步作为 Dify 的上游拆解和指令生成者，当前容易出现“编造不符项目情况”的危险行为（例如随意脑补项目综述）。必须修改 `main.py` 中下发给 Flash 的预处理 Prompt，强调“不可僭越”、“审慎是最重要的”，宁可模棱两可也不要犯错编造事实。
3. **消除过度加粗，回归正式公文排版**：目前生成的文本里有太多 Markdown 的加粗（`**`），这不符合真实招标/投标文件的严肃性。需要在 Flash 预处理 Prompt 和 Dify 执行 Prompt 双向发力，严格限制 Markdown 粗体的使用频率。
4. **修复表格生成空白问题**（当前最高优先级）：
   - **排查结论**：前端渲染正常但 Word 中表格空白，是因为 `docx_builder.py` 解析 Mistune 3 语法树时存在缺陷。Mistune 3 的 `table_head` 节点直接包含 `table_cell`（不再包裹 `table_row`），导致当前代码的解析循环直接跳过了整个表头，进而导致后续行数列数错乱，在 Word 中渲染为空白框。
   - **行动计划**：重写 `docx_builder.py` 中 `ctype == "table"` 的解析逻辑，分别适配 `table_head`（直接提取 cell）和 `table_body`（先提取 row 再提取 cell），确保所有行列数据被正确推入 `all_rows`。
5. **修复 Mermaid 图表渲染超时问题**（当前最高优先级）：
   - **排查结论**：为了解决大图表 URL 超长问题，我们已将其改为 POST 请求。但本地测试发现向 `https://kroki.io` 发送 `requests.post` 会报 `Read timed out` 错误。这是因为官方公共 API 对默认 Python `requests` 客户端可能有拦截、限流，或者其 JSON POST 格式存在兼容问题。
   - **行动计划**：
     - 首先尝试在请求中增加合法的 `User-Agent` 与超时重试机制。
     - 若仍不稳定，改为 `POST /mermaid/png` 发送 `text/plain` 纯文本格式。
     - 最优解：在未来的 `docker-compose` 中直接引入一个本地的 Kroki 容器，彻底消除外部网络依赖和限流风险。
