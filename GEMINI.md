# Project Guidelines - Bid

本文件记录了本项目的核心开发准则和协作流程。所有开发活动必须严格遵守以下规定：

## 核心协作准则

1. **技术选型与参考（最高优先级）**：严禁仅凭历史经验或训练数据进行决策。在采取任何行动（如配置 CI/CD、编写 Dockerfile、使用框架 API）之前，**必须**先查阅并引用相关技术栈的 **最新官方文档**（如 GitHub Actions, Docker, Next.js, FastAPI 等），确保方案符合当前版本的生产实践。
2. **语言规范**：始终使用 **中文** 进行沟通与对话。
3. **决策流程**：在进行任何具体的代码调整或实施之前，必须先详细说明接下来的实现方案。只有在用户确认方案后，方可进入执行阶段。用户对项目的方向拥有绝对掌控权。

## 开发流程

- **调研 (Research)**：系统地分析代码库，验证假设。对于新功能或复杂问题，优先使用最新官方文档。
- **策略 (Strategy)**：基于调研结果制定方案，并提交给用户审核。
- **执行 (Execution)**：在方案获批后，按照“计划 -> 行动 -> 验证”的循环进行迭代开发。
- **环境标准**：项目统一使用 **Docker Compose** 作为标准开发、调试与生产交付环境，确保环境的一致性。
- **质量保障 (Quality Assurance)**：
    1. **单元测试**：每个阶段性进展必须编写对应的单元测试，并确保执行通过。
    2. **用户验收 (UAT)**：完成代码验证后，必须提示用户进行验收测试，针对改动的功能点进行确认。
- **验证 (Validation)**：所有更改必须经过严格验证，包括但不限于自动化测试和 lint 检查。验收通过后方可进入下一阶段。

## 安全与规范

- 严禁泄露任何敏感信息（API Keys, Secrets 等）。
- 遵循现有的代码风格和架构模式。
- 优先选择简单、主流的技术方案。

## API 使用规范

- **Gemini API**：本项目统一使用最新版 `google-genai` SDK（v0.3.0+，全面替代旧版 `google-generativeai`）。在进行任何涉及 Gemini 模型交互的代码编写或重构之前，**必须**首先查阅并严格遵循项目内的参考文档：`docs/gemini_api_reference.md`。严禁凭借记忆使用过时的语法（如在异步流式生成器上错误使用 `await`），所有的客户端初始化、异步流式传输和配置结构均以此文档为唯一准则。
